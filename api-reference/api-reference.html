Relancy Integration API — Tenant Reference

Contract‑driven, audit‑first REST API for integrating your internal systems with the Relancy Operational Risk OS.

1. Overview

Style: JSON over HTTPS

Base URL (example): https://api.relancy.com/v1

Versioning: Path‑based (/v1, /v2, …)

Platform doctrine: Tenants cannot change schema, add fields, or run code inside Relancy. All access is contract‑driven and auditable.

Use this API to read cases, evidence metadata, risk signals, and training state, and to write case notes, status transitions (where allowed), and sales‑system sync acknowledgements.

2. Authentication & Required Headers

2.1 OAuth2 client credentials

Relancy uses OAuth2 client credentials with tenant‑scoped tokens.

  • Grant type: client_credentials
  • Audience: Relancy API
  • Scope: Provided during onboarding

Token endpoint (example):

POST https://auth.relancy.com/oauth2/token
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials&
client_id=YOUR_CLIENT_ID&
client_secret=YOUR_CLIENT_SECRET&
scope=relancy.api

Response:

{
  "access_token": "eyJhbGciOi...",
  "token_type": "Bearer",
  "expires_in": 3600
}

2.2 Required headers

All API calls must include:

  • Authorization: Bearer {access_token}
  • X-Relancy-Tenant-ID: {tenantId}
  • X-Relancy-User-ID: {userId} (the human or system identity on your side)
  • X-Relancy-Correlation-ID: {uuid} (trace across your systems and Relancy)
  • Idempotency-Key: {uuid} (required for all write operations)

Example headers:

X-Relancy-Tenant-ID: t_12345
X-Relancy-User-ID: u_98765
X-Relancy-Correlation-ID: 9f8c1b1a-...
Idempotency-Key: 4b2d7c3e-...

3. Core Resources & Endpoints

3.1 Cases

Use cases: Retrieve and track cases, lifecycle, and status; attach notes and evidence metadata.

GET /v1/cases

List cases with basic filters.

GET /v1/cases?status=open&createdSince=2026-01-01T00:00:00Z

Query parameters (examples):

  • status — open | closed | escalated | monitoring
  • createdSince — ISO 8601 timestamp
  • createdUntil — ISO 8601 timestamp
  • page — page number
  • pageSize — items per page (max 100)

Response (200):

{
  "items": [
    {
      "caseId": "c_001",
      "externalReference": "ORDER-12345",
      "status": "open",
      "type": "suspected_diversion",
      "priority": "high",
      "createdAt": "2026-04-02T20:14:00Z",
      "updatedAt": "2026-04-02T20:20:00Z"
    }
  ],
  "page": 1,
  "pageSize": 50,
  "total": 1
}

GET /v1/cases/{caseId}

Retrieve full case details.

GET /v1/cases/c_001

Response (200):

{
  "caseId": "c_001",
  "externalReference": "ORDER-12345",
  "status": "open",
  "type": "suspected_diversion",
  "priority": "high",
  "createdAt": "2026-04-02T20:14:00Z",
  "updatedAt": "2026-04-02T20:20:00Z",
  "summary": "Customer reports non-receipt; CCTV shows delivery to neighbor.",
  "assignee": {
    "userId": "inv_001",
    "displayName": "Investigator One"
  }
}

POST /v1/cases/{caseId}/notes

Add an internal note to a case.

POST /v1/cases/c_001/notes
Content-Type: application/json

{
  "text": "Package delivered to neighbor at 12:14. Customer unaware.",
  "visibility": "internal"
}

Response (201):

{
  "noteId": "n_001",
  "createdAt": "2026-04-02T20:25:00Z"
}

POST /v1/cases/{caseId}/status-transitions

Request a status change using the allowed contract.

POST /v1/cases/c_001/status-transitions
Content-Type: application/json

{
  "fromStatus": "open",
  "toStatus": "closed",
  "reason": "resolved_refund_issued"
}

Response (200):

{
  "caseId": "c_001",
  "status": "closed",
  "closedAt": "2026-04-02T21:00:00Z"
}

If the transition is not allowed by your contract, the API returns 409 Conflict.

GET /v1/cases/{caseId}/evidence

GET /v1/cases/c_001/evidence

Response (200):

{
  "items": [
    {
      "evidenceId": "e_001",

      "hash": "sha256:...",
      "createdAt": "2026-04-02T20:16:00Z"
    }

}

  "capturedAt": "2026-04-02T12:14:00Z",
  "description": "Front door camera, delivery event.",
  "uri": "https://your-storage.example.com/evidence/e_001",
  "hash": "sha256:..."
}

Response (201):

{
  "evidenceId": "e_001",
  "createdAt": "2026-04-02T20:16:00Z"

3.5 Sales-system alignment

These endpoints sync your sales/order system with Relancy’s risk and case view.

POST /v1/sales/orders/sync

Register or update an order snapshot for alignment.

POST /v1/sales/orders/sync
Content-Type: application/json

{
  "orderId": "ORDER-12345",
  "customerId": "CUST-001",
  "orderDate": "2026-04-02T11:45:00Z",
  "totalAmount": 129.99,
  "currency": "USD",
  "shippingAddressRegion": "IN-47130",
  "channel": "ecommerce",
  "status": "shipped"
}

Response (202):

{
  "orderId": "ORDER-12345",
  "acceptedAt": "2026-04-02T20:10:00Z"
}

POST /v1/sales/orders/{orderId}/acknowledgements

Acknowledge that your system has processed a Relancy event (for example, a flag or case creation).

POST /v1/sales/orders/ORDER-12345/acknowledgements
Content-Type: application/json

{
  "eventId": "evt_abc123",
  "status": "processed",
  "processedAt": "2026-04-02T20:30:00Z"
}

4. Webhooks (High-Level)

Relancy uses one‑way webhooks to notify your systems of key events. You respond by calling the Integration API.

Typical event types:

  • case.created
  • case.status.changed
  • evidence.added
  • risk.signal.detected
  • training.assignment.issued
  • sales.order.flagged
  • sales.order.fulfillment.updated

Webhook payload example:

{
  "eventId": "evt_abc123",
  "eventType": "case.created",
  "tenantId": "t_12345",
  "correlationId": "9f8c1b1a-...",
  "occurredAt": "2026-04-02T20:14:00Z",
  "payload": {
    "caseId": "c_001",
    "externalReference": "ORDER-12345",
    "status": "open"
  },
  "retryCount": 0,
  "signature": "v1=..."
}

Delivery and verification:

  • Non‑2xx responses trigger retries with exponential backoff.
  • Each webhook is signed using HMAC with a shared secret.
  • You should validate the signature, use eventId for idempotency, and call the API (for example, GET /v1/cases/{caseId}) if you need more detail.

5. Error Model

Relancy uses standard HTTP status codes with a consistent error body.

Error response example:

{
  "errorCode": "CASE_STATUS_TRANSITION_NOT_ALLOWED",
  "message": "Transition from 'closed' to 'open' is not allowed for this tenant.",
  "correlationId": "9f8c1b1a-...",
  "details": {
    "fromStatus": "closed",
    "toStatus": "open"
  }
}

Common status codes:

  • 400 Bad Request — invalid input
  • 401 Unauthorized — invalid or missing token
  • 403 Forbidden — not allowed for this tenant
  • 404 Not Found — resource does not exist
  • 409 Conflict — violates contract (for example, invalid status transition)
  • 429 Too Many Requests — rate limit exceeded
  • 500 Internal Server Error — unexpected error

6. Idempotency & Rate Limits

6.1 Idempotency

All write endpoints require Idempotency-Key.

  • Reusing the same key with the same payload returns the original result.
  • Reusing the same key with a different payload returns 409 Conflict.

6.2 Rate limits

Rate limits are enforced per tenant and per client.

  • Exceeding limits returns 429 Too Many Requests.
  • A Retry-After header indicates when you may retry.
