Relancy Webhook Registration API

Register, validate, and manage webhook endpoints for receiving event notifications from the Relancy Operational Risk OS.

1. Overview

The Webhook Registration API allows your systems to subscribe to Relancy event streams such as case lifecycle changes, evidence updates, risk signals, and sales‑system alignment events.

Webhook delivery is:

  • Explicit — tenants must register endpoints
  • Secure — all events are signed with a shared secret
  • Isolated — each tenant has its own configuration
  • Auditable — all registrations are logged and versioned

Base URL (example): https://api.relancy.com/v1

2. Authentication & Required Headers

2.1 OAuth2 client credentials

All webhook registration endpoints require a valid OAuth2 access token.

2.2 Required headers

  • Authorization: Bearer {access_token}
  • X-Relancy-Tenant-ID: {tenantId}
  • X-Relancy-User-ID: {userId}
  • X-Relancy-Correlation-ID: {uuid}
  • Idempotency-Key: {uuid} (for write operations)

3. Webhook Registration Endpoints

3.1 Create a webhook registration

POST /v1/webhooks/registrations

Registers a new webhook endpoint for your tenant. A validation challenge is sent before activation.

POST /v1/webhooks/registrations
Content-Type: application/json

Request body:

{
  "url": "https://your-system.example.com/relancy/webhooks",
  "sharedSecret": "your-generated-secret",
  "eventTypes": [
    "case.created",
    "case.status.changed",
    "evidence.added",
    "risk.signal.detected"
  ],
  "retryPolicy": {
    "maxAttempts": 12,
    "initialBackoffSeconds": 5
  }
}

Response (202 Accepted):

{
  "registrationId": "wh_001",
  "status": "pending_validation",
  "createdAt": "2026-04-02T20:14:00Z"
}

Relancy will immediately send a challenge request to the provided URL. The webhook becomes active only after successful validation.

3.2 Validate a webhook (challenge handshake)

When you register a webhook, Relancy sends a challenge request to verify ownership and ensure your endpoint can receive signed events.

Challenge request (Relancy → Tenant)

POST {yourWebhookUrl}
Content-Type: application/json
Relancy-Signature: v1=...

{
  "challenge": "c_9f8c1b1a",
  "timestamp": "2026-04-02T20:14:00Z"
}

Your required response:

{
  "challenge": "c_9f8c1b1a"
}

If the challenge is not echoed back correctly within the timeout window, the webhook remains inactive.

3.5 Update a webhook registration

PATCH /v1/webhooks/registrations/{registrationId}

PATCH /v1/webhooks/registrations/wh_001
Content-Type: application/json

Request body (example):

{
  "eventTypes": [
    "case.created",
    "case.status.changed",
    "evidence.added"
  ]
}

Response (200):

{
  "registrationId": "wh_001",
  "status": "active",
  "updatedAt": "2026-04-02T21:00:00Z"
}

3.6 Delete a webhook registration

DELETE /v1/webhooks/registrations/{registrationId}

DELETE /v1/webhooks/registrations/wh_001

Response (204 No Content)

4. Webhook Delivery Model

Once a webhook is active, Relancy delivers events using signed POST requests.

4.1 Event payload example

{
  "eventId": "evt_abc123",
  "eventType": "case.created",
  "tenantId": "t_12345",
  "correlationId": "9f8c1b1a-...",
  "occurredAt": "2026-04-02T20:14:00Z",
  "payload": {
    "caseId": "c_001",
    "externalReference": "ORDER-12345",
    "status": "open"
  },
  "retryCount": 0,
  "signature": "v1=..."
}

4.2 Retry behavior

  • Non‑2xx responses trigger exponential backoff
  • Retries stop after maxAttempts
  • Failed deliveries may be sent to a tenant‑scoped dead‑letter queue (if enabled)

4.3 Signature verification

All webhook requests include a Relancy-Signature header containing an HMAC signature using your shared secret.

You must:

  • Compute the HMAC using the request body
  • Compare it to the signature header
  • Reject the request if the signature does not match

5. Error Model

Error response example:

{
  "errorCode": "WEBHOOK_VALIDATION_FAILED",
  "message": "The provided endpoint did not return the expected challenge response.",
  "correlationId": "9f8c1b1a-..."
}

Common status codes:

  • 400 Bad Request — invalid input
  • 401 Unauthorized — invalid or missing token
  • 403 Forbidden — not allowed for this tenant
  • 404 Not Found — registration does not exist
  • 409 Conflict — duplicate registration or invalid state
  • 500 Internal Server Error — unexpected error
