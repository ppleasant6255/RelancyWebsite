Numbering Algorithm Configuration

Relancy Platform Knowledge Base

1. Purpose

This article defines how Tenant Admins configure numbering algorithms for cases, incidents, and evidence within each Business Unit. Numbering algorithms ensure consistent, traceable, and business‑aligned identifiers across the platform.

2. Role Authority

Only Tenant Admins may view or modify numbering algorithm settings. Directors, Supervisors, Investigators, and all other roles are restricted from accessing these configuration controls.

All changes are logged with timestamp, actor, Business Unit, and full before/after configuration details.

3. Scope of Configuration

Tenant Admins may configure numbering algorithms for the following record types:

  • Case Numbers
  • Incident Numbers
  • Evidence Numbers

Each Business Unit may define its own independent numbering scheme. Changes apply only to future records and do not modify existing identifiers.

4. Algorithm Structure

A numbering algorithm is composed of ordered segments. Segments define how the final identifier is constructed. Tenant Admins may add, remove, reorder, and configure segments as needed.

Supported Segment Types

  • Static Text: Fixed text such as “HR”, “OPS”, or “SEC”.
  • Business Unit Code: The code assigned to the Business Unit.
  • Date Components: YYYY, YY, MM, DD, or combinations such as YYYYMM.
  • Sequence Number: Auto‑incrementing numeric value with configurable padding.
  • Tenant Code: The tenant’s unique identifier.
  • Region or Location Code: Optional geographic or organizational segment.
  • Custom Segment: Admin‑defined text or token.

Segment Operations

  • Add new segments
  • Remove existing segments
  • Reorder segments
  • Configure segment properties such as padding, separators, and casing

5. Validation Rules

  • Algorithms must contain at least one dynamic segment (date or sequence).
  • Algorithms must not produce duplicate identifiers.
  • Sequence segments must define starting value, padding length, and reset behavior.
  • Static text segments must be between 1 and 20 characters.
  • Custom segments must be explicitly named and documented.

6. Preview and Testing

Before saving changes, Tenant Admins must generate a preview of the next five identifiers. This ensures the algorithm behaves as expected and aligns with business requirements.

Admins must acknowledge that changes may affect downstream workflows and reporting structures.

7. Audit Requirements

All changes to numbering algorithms require a justification of at least 10 characters. The audit log records:

  • Admin identity
  • Timestamp
  • Business Unit affected
  • Full before/after configuration
  • Justification text

8. Enforcement Model

  • All numbering is generated server‑side.
  • Client‑side overrides are rejected.
  • Algorithms cannot retroactively modify existing identifiers.
  • Changes apply only to new records created after the update.
