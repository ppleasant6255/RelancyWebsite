How Relancy Isolates Tenant Data

Shared API, isolated databases, and strict boundaries.

Back to Knowledge Base

Overview

Relancy uses a classic, proven multi-tenant pattern: a shared API layer with isolated tenant data. This architecture allows the platform to enforce a consistent operational worldview while keeping each tenant’s data, audit history, and storage fully separated.

Shared API: shared worldview

The API layer represents Relancy’s operational contract—how entities behave, how lifecycle events are enforced, and how audit is recorded. All tenants use the same API surface so that:

  • Lifecycle behavior is consistent
  • Security and privacy rules are uniformly applied
  • New features and fixes are delivered to everyone
  • The platform remains predictable and supportable

In short, the API is shared because the rules are shared.

Isolated databases: isolated risk

Behind the shared API, each tenant’s data is stored in an isolated database or schema boundary. This means:

  • No cross-tenant reads or writes
  • Separate audit logs per tenant
  • Independent backup and retention policies
  • Clear legal and operational separation

Your data is never co-mingled with other tenants’ data at the database level.

How routing works

Every request to the Relancy API carries tenant context (for example, a tenant identifier and authenticated user identity). The platform uses this context to:

  • Select the correct tenant database or schema
  • Apply tenant-specific isolation rules
  • Scope audit entries to the correct tenant
  • Enforce authorization and access control

The routing is invisible to end users but central to how Relancy maintains strict separation.

Why this matters

This architecture provides several important benefits:

  • Security: One tenant cannot access another tenant’s data.
  • Compliance: Data boundaries are clear and enforceable.
  • Reliability: Issues in one tenant’s data do not corrupt others.
  • Scalability: Tenant databases can be scaled and managed independently.

Dedicated infrastructure in Private Deployment

In the Private Deployment tier, this isolation is taken even further: tenants receive dedicated databases, dedicated API clusters, and dedicated storage. The same principles apply, but the infrastructure is physically and logically separated from the shared SaaS environment.
